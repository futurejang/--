# 작업 프로세스

> ⚠️ **추측을 사실처럼 말하지 말 것!** 모든 가설은 반드시 검증 후 결론.
> ⛔ **코드 작성 전 반드시 4단계까지 완료하고 사용자 승인을 받을 것.**

## 1단계: 문제/요청 이해

- 문제 현상을 명확히 기술
- 불분명한 부분이 있으면 사용자에게 질문
- "~일 것 같습니다"가 아니라 실제 코드를 확인

## 2단계: 원인 분석 (문제 해결의 경우)

- 가설 수립 → 가설 검증 → 원인 확정
- ❌ "이게 원인입니다" (검증 없이)
- ✅ "가설: ~일 수 있습니다. 검증해보겠습니다." → "확인 결과, ~가 원인입니다"

## 3단계: 해결책 탐색

- 해결 방안 2-3개 제시, 각 방안의 영향 범위 분석
- 사전 검증 가능하면 검증

## 4단계: 작업 계획 보고 (코드 작성 전 필수!)

> ⛔ 사용자가 "그냥 빨리 해줘"라고 해도, 이 보고를 먼저 하세요.

```
📋 작업 계획 보고

🔍 문제 상황 — 어떤 상황에서 어떤 증상이 발생하는지
🎯 목표 — 작업 후 어떤 상태가 되어야 하는지
🔬 원인 — 검증된 원인만. 추측은 "가설"로 명시

📁 변경 예정 파일
| 파일 경로 | 변경 내용 | 비고 |

⚡ Before → After
🎨 사용할 토큰 / 재사용할 컴포넌트 / 새로 필요한 토큰

이대로 진행해도 될까요?
```

## 5단계: 작업 실행

- 승인받은 계획대로 진행
- 예상치 못한 상황 → 중단 후 보고

## 6단계: 결과 검증

루트 CLAUDE.md의 규칙 위반 여부를 확인한다:

| # | 확인 항목 | 검증 방법 |
|---|----------|----------|
| 1 | 빌드 정상 | `npm run build` 통과 |
| 2 | 루트 CLAUDE.md 토큰 규칙 준수 | hex, rgb, Tailwind 기본값, 토큰 외 spacing 없는지 |
| 3 | 기존 컴포넌트 재사용 | 대체 가능한 기존 컴포넌트를 놓치지 않았는지 |
| 4 | 기존 기능 정상 | 기존 컴포넌트/페이지 깨지지 않았는지 |

## 7단계: 작업 완료

- 6단계 검증 **전부 통과** 후에만 "완료" 선언
- 변경 사항 요약 보고

## 금지 사항

| 금지 | 올바른 대안 |
|------|-----------|
| 허락 없이 새 파일/컴포넌트 생성 | 사용자에게 먼저 제안 후 승인 |
| 기존 아키텍처 임의 변경 | 변경 필요 시 이유와 함께 제안 |
| 요청 범위 밖 리팩토링 | "이 부분도 개선하면 좋겠는데, 할까요?" |
| 문제 발견 시 바로 수정 | 보고 → 해결책 2-3개 제시 → 승인 후 수정 |
